## El problema

Ya no trabajas en una sola herramienta de IA вҖ” hoy Claude Code, maГұana Cursor o Windsurf,
Conductor para trabajo en paralelo. Pero **el contexto no viaja contigo**: cada
herramienta, proyecto y sesiГіn empieza de cero, asГӯ que reвҖ‘explicas el mismo cГіdigo una y
otra vez. Encima cada agente **quema tokens** con contexto inflado, **olvida** todo entre
sesiones y tiende a **sobreвҖ‘diseГұar** вҖ” y lo poco que vale la pena recordar nunca se
captura. Y dejar el arreglo listo a mano es engorroso y hay que repetirlo en cada Mac.

## La soluciГіn

Una capa local que hace que tu contexto **se traspase** (*carry over*) entre herramientas,
proyectos y sesiones:

- рҹ§  **Memoria persistente y compartida** entre herramientas, repos y sesiones вҖ” un store
  local que es tuyo, con **recall semГЎntico** (por significado) con scope a cada repo o a un
  grupo de repos relacionados.
- рҹ•ё **Conocimiento estructurado** вҖ” facts + entidades tipadas + relaciones (un grafo
  consultable), indexado por el repo del que viene.
- рҹ”Ғ **AutoвҖ‘recall** вҖ” al iniciar una sesiГіn en un repo, se inyecta como contexto lo que
  carryover sabe de ese repo (~0.5s, ~500 tokens, comprimido). MГЎs `/recall` / `co-recall` a demanda
  (solo el repo actual; `--all` para todos los repos).
- рҹ“Ҡ **Dashboard local** (`co-dash`) вҖ” explora, busca, filtra y **gestiona** (borra/limpia)
  tu conocimiento, un grafo de relaciones, y tus wikis.
- рҹ“„ **Wiki automГЎtica** вҖ” un LLM escribe docs + diagramas mermaid (visiГіn general, arquitectura,
  flujos + un catГЎlogo de **Features**) y los actualiza de forma **incremental** (preserva las
  pГЎginas existentes, aГұade solo lo que cambiГі) al hacer push a master/main.
- рҹ’ҫ **Pregunta de "guardar lo importante"** al final В· рҹ©ә **`carryover doctor`** В·
  toggle de routing **on/off** В· `carryover wrap <tool>` para Cursor/Codex/вҖҰ
- Barra de estado **рҹҗҙ/рҹ§ **, slash commands, aliases de terminal.

Un instalador idempotente, **100% local**, **standalone por defecto** вҖ” con integraciones
opcionales de headroom (store compartido + compresiГіn de tokens) y ponytail вҖ” para que tu
contexto te siga en vez de reiniciarse.

### Configurarlo en cada herramienta (la parte cross-tool)

La memoria y la compresiГіn se comparten a travГ©s del proxy local вҖ” cada herramienta solo
necesita que la apuntes a Г©l **una vez**. (El proxy debe estar corriendo: revГӯsalo con
`carryover doctor` o `hr-status`.)

- **Claude Code / Conductor** вҖ” automГЎtico. El `install.sh` ya dejГі el routing; nada que hacer.
- **Cursor** вҖ” corre `carryover wrap cursor` (delega en `headroom wrap cursor`). Imprime la
  config exacta para pegar en los ajustes de modelo de Cursor (apunta su API base al proxy).
  Desde ahГӯ, la IA de Cursor comparte la misma memoria + compresiГіn que Claude.
- **Windsurf / Devin** вҖ” en sus ajustes de IA/modelo, pon una **OpenAI base URL** personalizada:
  `http://127.0.0.1:8787/v1`. (`carryover wrap windsurf` imprime este recordatorio.)
- **Codex / Aider / Cline / Continue / Goose / Copilot** вҖ” `carryover wrap <tool>`.
- **Cualquier otra herramienta compatible con OpenAI/Anthropic** вҖ” apunta su base URL al proxy:
  `http://127.0.0.1:8787` (estilo Anthropic) o `http://127.0.0.1:8787/v1` (estilo OpenAI).

Dos cosas a tener en cuenta:

- Los **comandos de terminal** (`co-dash`, `mem-save`, `co-recall`, `carryover вҖҰ`) funcionan en
  la terminal integrada de **cualquier** herramienta вҖ” viven en tu shell (`~/.zshrc`), no en una app.
- Los **slash commands** (`/headroom`, `/recall`, `/carryover`, `/wiki-enable`) y la barra рҹҗҙ/рҹ§ 
  son **solo de Claude Code**. Las demГЎs herramientas reciben la memoria + compresiГіn, no el slash UI.
- Una herramienta **no** se auto-detecta вҖ” comparte todo solo despuГ©s de apuntarla al proxy una vez.

## Instalar

```bash
git clone https://github.com/Cfvillarroel/carryover.git ~/carryover
bash ~/carryover/install.sh
```

Instala **carryover standalone** вҖ” memoria, recall, wikis y el dashboard, con un store local
propio (`~/.carryover/memory.db`). Abre una terminal nueva (o `source ~/.zshrc`) y reinicia
Claude. Requisitos: macOS В· Python 3 В· Claude Code (`claude`) en el PATH.

**Integraciones opcionales** (opt-in):
```bash
bash ~/carryover/install.sh --with-headroom   # + headroom: proxy de compresiГіn + backend de memoria compartido
bash ~/carryover/install.sh --with-ponytail   # + ponytail: plugin lazy-dev de Claude
bash ~/carryover/install.sh --full            # carryover + headroom + ponytail
```
- **headroom** convierte la memoria en un store *compartido* entre herramientas y aГұade el
  proxy de compresiГіn (routing para Claude / Cursor / CodexвҖҰ). Si estГЎ instalado, carryover lo
  usa solo; si no, usa el store propio. Necesita `brew install python@3.13` (la extensiГіn
  Rust/PyO3 de headroom no compila en 3.14).
- **ponytail** es un plugin de terceros independiente вҖ” puramente opcional.

El instalador es **idempotente**: symlinkea los hooks, el dashboard, la barra y el comando
`/headroom`, y aГұade los aliases a `~/.zshrc`.

**ВҝNo usas Claude Code?** carryover no es solo para Claude. El proxy de headroom es
compartido, asГӯ que la memoria + compresiГіn aplican a **cualquier** herramienta una vez
corriendo вҖ” Cursor, Windsurf, Codex, AiderвҖҰ solo apuntas cada una al proxy (mira
[Funciona con](#funciona-con) para el one-liner por herramienta). Los extras especГӯficos
de Claude del `install.sh` (barra de estado, `/headroom`, los plugins) son para Claude
Code / Conductor; lo demГЎs comparte el mismo proxy + memoria.

### Instalar solo el plugin de Claude (opcional)

## Alcance: global vs porвҖ‘repo

- **Global (una vez por Mac):** proxy + memoria + config de Claude. Global **por diseГұo** вҖ”
  es lo que hace que el contexto sea compartido entre todos los repos y herramientas.
- **Por repo / un conjunto:** la **wiki** вҖ” corre `co-wiki-enable` en cada repo que quieras
  (uno, varios o todos). Instala un hook `pre-push` solo ahГӯ.
- La memoria es global pero **con scope por repo**: recall devuelve por defecto las memorias
  del repo actual. Agrupa repos relacionados (ej. el front + back de un producto) para que
  compartan recall, listГЎndolos en una lГӯnea de `~/.carryover/groups.conf` (separados por espacio/coma).

## Mensajes entre workspaces

ВҝCorres varios workspaces de Conductor a la vez? Comparten un Гәnico store de memoria, asГӯ que
pueden dejarse notas entre sГӯ вҖ” "mergeГ© X, rebasea", "build roto, no hagas pull". Cada workspace
responde a varios nombres: su **codename** (la carpeta del worktree, p.ej. `paris`, `sarajevo`) вҖ”
la **direcciГіn estable y duradera**; su **nombre de workspace** (la etiqueta que muestra Conductor,
que reescribe al nombre del branch/tarea cuando interactГәas con Г©l la primera vez); y su **nombre de
proyecto** (el encabezado bajo el que vive, p.ej. `proyectate-back`). Manda al **nombre de proyecto**
para llegar a cualquier workspace de ese proyecto вҖ” ideal para un par frontendвҶ”backend entre repos вҖ”
o al **codename** para uno especГӯfico. Todos son globales, no atados a un repo. Direcciona por
**codename o nombre de proyecto** (ambos estables ante renombrados); un tГӯtulo que Conductor ya
renombrГі puede no resolver.

- `co-send <nombre> <mensaje>` вҖ” dejar una nota a otro workspace o proyecto entero. `all` hace broadcast.
- `co-inbox` вҖ” leer las notas dirigidas a **este** workspace (mГЎs los broadcasts). Leerlas las
  consume para que no se repitan; `co-inbox --peek` lee sin consumir.
- `co-connect <nombre>` вҖ” vincular dos workspaces/proyectos en ambos sentidos (persistente). Luego
  `co-say <mensaje>` envГӯa a **todos** los conectados sin reteclear nombres; `co-connections` los
  lista y `co-disconnect <nombre>` desconecta. Гҡtil para un par frontend/backend entre repos.
- `/handoff <nombre>` (en el chat) вҖ” traspasar una tarea: el agente escribe un resumen (quГ© se hizo,
  quГ© falta, archivos clave) y lo manda a ese workspace/proyecto. TambiГ©n se sugiere solo cuando un
  turno menciona un workspace conectado.
- `/handover <nombre>` (en el chat; en shell `co-handover`) вҖ” como `/handoff`, pero una **transferencia
  activa**: el destino ademГЎs recibe una notificaciГіn de escritorio y, cuando se le entrega el inbox,
  se le dice al agente que **ejecute la tarea ahora**, no que solo la lea. Conductor no puede despertar
  un workspace inactivo, asГӯ que la notificaciГіn es tu seГұal para abrirlo; ahГӯ la tarea corre en su
  primer turno.
- **Entrega automГЎtica:** las notas pendientes se inyectan en el contexto del workspace cuando su
  sesiГіn **arranca**, **antes de cada uno de tus turnos** (`UserPromptSubmit`), y **cuando el agente
  termina un turno** (`Stop`) вҖ” asГӯ una nota que llega mientras trabaja se recoge en cuanto queda
  libre, sin interrumpirlo a media tarea. Se entrega una sola vez y se marca leГӯda. La entrega por
  `Stop` tiene un freno anti-bucle (`stop_hook_active`) para que dos workspaces conectados no entren
  en ping-pong. (Sigue siendo pull: un agente totalmente inactivo вҖ”que ya te devolviГі el controlвҖ”
  la recibe en su siguiente turno.)

Solo pull, por diseГұo: a un agente *en marcha* no se le interrumpe вҖ” las notas llegan cuando su
sesiГіn vuelve a arrancar, o cuando ejecutas `co-inbox`. Los mensajes viven en su propio buzГіn
`@<workspace>`, asГӯ que nunca aparecen en el recall normal ni ensucian el conocimiento de un repo.

**Cada workspace lo tiene automГЎticamente.** Los comandos, los hooks de entrega y el buzГіn compartido
se instalan una vez a nivel de usuario (con `install.sh`, o con el plugin de Claude), asГӯ que cualquier
workspace nuevo de Conductor вҖ” incluido un worktree reciГ©n creado вҖ” puede enviar y recibir sin
configuraciГіn por workspace; su direcciГіn sale del entorno de Conductor. La Гәnica excepciГіn es
`co-connect`/`co-say`: un workspace nuevo no tiene enlaces hasta que lo conectas una vez (los dirigidos
`co-send`/`/handoff`/`/handover` no necesitan ninguno).

**PruГ©balo** вҖ” dos workspaces del mismo proyecto, `paris` y `surabaya`:


### Conectar workspaces (p.ej. frontend вҶ” backend, incluso entre repos)

Vincula dos workspaces una vez, y luego habla sin reteclear nombres largos:

```sh

co-say "el contrato de /users cambiГі: email ahora es obligatorio"   # вҶ’ a todos los conectados
co-connections                       # рҹ”— connected to: proyectate-back-main
```

El workspace de backend lo recibe como cualquier otra nota (al arrancar, en su prГіximo turno, o con
`co-inbox`). `co-disconnect proyectate-back-main` desconecta.

(Los comandos nuevos llegan con `carryover update`; hasta entonces un workspace puede usarlos desde
su checkout: `python3 claude/hooks/co-mem send <ws> "<msg>"`.)

## Playbooks (`!macros`)

Procedimientos reusables que disparГЎs escribiendo `!nombre` en cualquier prompt de Claude вҖ” como los
playbooks de Devin. Un playbook es un archivo markdown en `~/.carryover/playbooks/`; cuando tu prompt
contiene `!grill`, un hook `UserPromptSubmit` inyecta ese archivo y el agente sigue el procedimiento.


## Wiki automГЎtica (local, formato GitHub Wiki)

Genera una wiki del proyecto con Claude headless (`claude -p`) вҖ” visiГіn general,

